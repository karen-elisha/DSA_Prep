# Top K Frequent Elements — Explained

## The Problem
Given an array `nums` and an integer `k`, return the `k` most frequent elements.

## Your Code (cleaned up, with imports)

```java
import java.util.HashMap;
import java.util.ArrayList;
import java.util.List;
import java.util.Map;
import java.util.Collections;

class Solution {
    public int[] topKFrequent(int[] nums, int k) {
        HashMap<Integer, Integer> map = new HashMap<>();
        for (int num : nums) {
            if (map.containsKey(num)) {
                map.put(num, map.get(num) + 1);
            } else {
                map.put(num, 1);
            }
        }

        List<Map.Entry<Integer, Integer>> list = new ArrayList<>(map.entrySet());
        Collections.sort(list, (a, b) -> b.getValue() - a.getValue());

        int[] ans = new int[k];
        for (int i = 0; i < k; i++) {
            ans[i] = list.get(i).getKey();
        }
        return ans;
    }
}
```

## Step-by-Step Walkthrough

Say `nums = [1, 1, 1, 2, 2, 3]` and `k = 2`.

**Step 1 — Count frequencies with a HashMap**

```
map = { 1: 3, 2: 2, 3: 1 }
```

Each key is a number from the array, each value is how many times it appeared.

**Step 2 — Turn the map into a sortable List**

**Step 3 — Sort that list by frequency (descending)**

**Step 4 — Grab the top `k` keys**

Now let's dig into steps 2 and 3, since that's the part most people find confusing.

---

## Why You Need a `List` Here

A `HashMap` in Java is fundamentally **unordered**. You cannot sort a `HashMap` directly — there's no `Collections.sort(map)` that works, because a map doesn't have a defined sequence, just key-value associations.

To sort something, Java needs a structure that has a well-defined order and supports random access/reordering — that's exactly what a `List` gives you.

So the trick is:

```java
List<Map.Entry<Integer, Integer>> list = new ArrayList<>(map.entrySet());
```

Let's break this line into pieces:

### `map.entrySet()`
Every `Map` has an `entrySet()` method that returns a `Set<Map.Entry<K, V>>` — a **view** of the map as a collection of key-value pairs bundled together as `Map.Entry` objects.

- `Map.Entry<Integer, Integer>` is just a small container object holding one key and one value together, with `getKey()` and `getValue()` methods to read them.
- For `map = {1: 3, 2: 2, 3: 1}`, `entrySet()` conceptually gives you: `{(1,3), (2,2), (3,1)}` — three `Entry` objects, order not guaranteed.

### `new ArrayList<>(map.entrySet())`
This is the **copy constructor** pattern. `ArrayList` has a constructor that accepts any `Collection` and copies its elements into a new, independent `ArrayList`.

So this line takes the entry set (a `Set`, unordered, and technically a "live view" of the map) and copies those entries into a fresh `ArrayList<Map.Entry<Integer,Integer>>` — a structure that:
1. Has indices (`list.get(0)`, `list.get(1)`, ...)
2. Can be freely reordered without touching the original map
3. Can be sorted with `Collections.sort`

**Why not just sort the Set directly?** Because `Set` doesn't guarantee order or provide index-based access — you can't call `.get(i)` on a `Set`. `List` is the interface that promises ordering and indexing, so it's the natural container for anything you intend to sort or iterate in a specific sequence.

---

## Why `Collections.sort` Works Here

`Collections` is a utility class full of static helper methods that operate on collections (`sort`, `reverse`, `max`, `min`, `shuffle`, etc.). It's not a collection itself — think of it as a toolbox.

```java
Collections.sort(list, (a, b) -> b.getValue() - a.getValue());
```

This has two arguments:

### 1. `list` — the thing being sorted
`Collections.sort(List<T> list)` mutates the list **in place**, rearranging its elements. It requires a `List`, not a `Set` or `Map` — this is exactly why Step 2 converted things into a `List` first.

### 2. The Comparator — `(a, b) -> b.getValue() - a.getValue()`

This is a **lambda expression** implementing the `Comparator<T>` functional interface, which has one abstract method:

```java
int compare(T a, T b);
```

The contract for `compare(a, b)`:
- **Negative** → `a` should come before `b`
- **Zero** → `a` and `b` are considered equal in order
- **Positive** → `a` should come after `b`

So `b.getValue() - a.getValue()`:
- If `b`'s frequency > `a`'s frequency → result is positive → `a` is placed *after* `b`
- If `a`'s frequency > `b`'s frequency → result is negative → `a` is placed *before* `b`

Net effect: **entries with higher frequency come first** — a descending sort by value.

If you had written `a.getValue() - b.getValue()` instead, you'd get an *ascending* sort (smallest frequency first) — the opposite of what this problem needs.

### Why a lambda instead of writing frequency into a `Comparable` class?
`Map.Entry` doesn't know anything about "sort by value" by default — it has no natural ordering. Rather than creating a custom class implementing `Comparable`, Java lets you pass the sorting logic directly as a lambda at the call site. This is cleaner and avoids extra boilerplate for a one-off sort.

## Putting It Together — Full Trace

For `nums = [1, 1, 1, 2, 2, 3]`, `k = 2`:

| Step | State |
|---|---|
| After counting | `map = {1:3, 2:2, 3:1}` |
| `entrySet()` | `{(1,3), (2,2), (3,1)}` (order not guaranteed) |
| `new ArrayList<>(...)` | `list = [(1,3), (2,2), (3,1)]` (some order, now indexable) |
| After `Collections.sort` | `list = [(1,3), (2,2), (3,1)]` — already descending by value in this example |
| Extract top `k=2` keys | `ans = [1, 2]` |

**Result:** `[1, 2]` — correct, since `1` appears 3 times and `2` appears 2 times.

## Complexity

- Building the HashMap: **O(n)**
- Converting to `ArrayList`: **O(m)** where `m` = number of distinct elements
- `Collections.sort`: **O(m log m)** (Java's sort uses TimSort — a hybrid merge sort, for objects)
- Extracting top `k`: **O(k)**

Overall: **O(n + m log m)**. 

**Note:** For large inputs, this is not the most optimal approach — a **bucket sort** (index = frequency, bucket = list of numbers with that frequency) or a **min-heap of size k** achieves **O(n)** or **O(n log k)** respectively. But your `Collections.sort` approach is correct, readable, and perfectly fine for interview settings unless the interviewer explicitly asks for a sub-O(n log n) solution.

## Key Takeaways
1. `HashMap.entrySet()` gives you key-value pairs as `Map.Entry` objects — but as an unordered `Set`.
2. Wrap that in `new ArrayList<>(...)` when you need indexing or sorting — `List` is the ordered, indexable cousin of `Set`.
3. `Collections.sort(list, comparator)` needs a `List` (not a `Set`/`Map`) and a `Comparator` lambda telling it *how* to compare two elements.
4. The comparator's return sign (negative/zero/positive) controls ordering — `b - a` for descending, `a - b` for ascending.
