# Valid Anagram — Solution Explanation

## Problem Statement
Given two strings `s` and `t`, return `true` if `t` is an anagram of `s`, and `false` otherwise.

An anagram is formed by rearranging the letters of one word to produce another, using all the original letters exactly once.

**Example 1:**
```
Input: s = "anagram", t = "nagaram"
Output: true
```

**Example 2:**
```
Input: s = "rat", t = "car"
Output: false
```

**Constraints:**
- `1 <= s.length, t.length <= 5 * 10^4`
- `s` and `t` consist of lowercase English letters only.

---

## Approach: Frequency Counting (Fixed-Size Array)

Since both strings consist only of lowercase English letters (`a`–`z`), we can use a fixed-size array of 26 integers to track the frequency of each character.

### Intuition
Two strings are anagrams of each other **if and only if** they contain exactly the same characters with exactly the same frequencies. So instead of sorting both strings (which costs `O(n log n)`), we can:

1. Count how many times each letter appears in `s` (increment).
2. Count how many times each letter appears in `t` (decrement).
3. If the strings are true anagrams, every count should cancel out back to **zero**.

This works because incrementing for `s` and decrementing for `t` in the *same* array means:
- If a letter appears the same number of times in both strings → net count = 0.
- If a letter appears a different number of times (or doesn't appear in one) → net count ≠ 0.

### Step-by-Step Walkthrough

**Step 1: Length Check (Early Exit)**
```java
if (s.length() != t.length())
    return false;
```
If the lengths differ, they can never be anagrams — no need to do any further work. This is an `O(1)` short-circuit that saves time on obviously invalid inputs.

**Step 2: Build the Frequency Array**
```java
int count[] = new int[26];
```
An array of size 26 represents the 26 lowercase letters `a` through `z`. Index `0` = `'a'`, index `1` = `'b'`, ..., index `25` = `'z'`.

**Step 3: Increment Counts for `s`**
```java
for (int i = 0; i < s.length(); i++) {
    count[s.charAt(i) - 'a']++;
}
```
For each character in `s`, we map it to an array index using `charAt(i) - 'a'`.
- Example: `'c' - 'a' = 2`, so `count[2]` is incremented.

After this loop, `count[i]` holds the frequency of the `i`-th letter in `s`.

**Step 4: Decrement Counts for `t`**
```java
for (int i = 0; i < t.length(); i++) {
    count[t.charAt(i) - 'a']--;
}
```
Same mapping, but this time we decrement. If `t` has the exact same letters in the exact same quantities as `s`, every increment from Step 3 gets perfectly cancelled out by a decrement here.

**Step 5: Verify All Counts Are Zero**
```java
for (int i = 0; i < count.length; i++) {
    if (count[i] != 0) {
        return false;
    }
}
```
If **any** index has a non-zero value, it means:
- Either `s` had more of that letter than `t` (count > 0), or
- `t` had more of that letter than `s` (count < 0)

Either case means the strings are **not** anagrams.

**Step 6: Return `true`**
If we pass through the entire count array without finding a non-zero value, all letters matched perfectly in frequency — the strings are valid anagrams.

## Full Annotated Code

```java
class Solution {
    public boolean isAnagram(String s, String t) {
        // Quick rejection: anagrams must have equal length
        if (s.length() != t.length())
            return false;
        else {
            int count[] = new int[26]; // frequency table for 'a' to 'z'

            // Tally character frequencies from s
            for (int i = 0; i < s.length(); i++) {
                count[s.charAt(i) - 'a']++;
            }

            // Subtract character frequencies from t
            for (int i = 0; i < t.length(); i++) {
                count[t.charAt(i) - 'a']--;
            }

            // If any letter's count didn't return to zero, mismatch found
            for (int i = 0; i < count.length; i++) {
                if (count[i] != 0) {
                    return false;
                }
            }
        }
        return true;
    }
}
```

## Complexity Analysis

| Metric | Value | Explanation |
|--------|-------|--------------|
| **Time Complexity** | `O(n)` | We make two linear passes over the strings (length `n`) plus one constant pass over a fixed 26-element array. Overall this is `O(n + 26) = O(n)`. |
| **Space Complexity** | `O(1)` | The `count` array always has a fixed size of 26 regardless of input size, so space usage doesn't grow with input — this is constant extra space. |

This is optimal for this problem — you cannot verify an anagram in less than `O(n)` time since every character must be examined at least once.

---

## Why This Beats the Sorting Approach

A common alternative solution is:
```java
char[] sArr = s.toCharArray();
char[] tArr = t.toCharArray();
Arrays.sort(sArr);
Arrays.sort(tArr);
return Arrays.equals(sArr, tArr);
```

| Approach | Time | Space |
|----------|------|-------|
| Sorting | `O(n log n)` | `O(n)` (for the char arrays) |
| Frequency Array (this solution) | `O(n)` | `O(1)` |

The frequency-counting approach is strictly better in both time and space for this problem, since we know the character set is limited to lowercase English letters.

---

## Edge Cases Handled

- **Different lengths** → immediately returns `false` (e.g., `s = "ab"`, `t = "a"`).
- **Same letters, different frequencies** → caught by the count mismatch (e.g., `s = "aab"`, `t = "abb"`).
- **Identical strings** → trivially returns `true` since every count cancels to zero.
- **Single character strings** → works correctly since loops just run once.

## Possible Extension (Unicode / Case-Insensitive Input)

If the problem allowed uppercase letters, digits, or Unicode characters, a `HashMap<Character, Integer>` would be needed instead of a fixed 26-size array, since the character range would no longer be limited to `'a'`–`'z'`. That would change space complexity to `O(k)`, where `k` is the number of distinct characters.

---

## Summary

This solution elegantly uses a **single shared frequency array** and the increment/decrement trick to check character frequency equality in one pass each, achieving optimal `O(n)` time and `O(1)` space — ideal for this constrained-alphabet problem.
